02

📜 التمهيد

يُعدّ هذا العقد التقني ملحقاً فنّياً مكمّلاً ومتمّماً للعقد التجاري الموقّع بين الطرفين بشأن تطوير نظام إدارة المعهد التدريبي المتكامل (IIMS)، ويُقرأ ويُفسَّر معه كوحدة واحدة.

يهدف هذا العقد إلى توضيح المواصفات التقنية الدقيقة، معايير الجودة، الأداء، الأمان، والصيانة، لضمان تسليم نظام يفي بأعلى المعايير المهنية.

1
المراجع والالتزامات

يستند هذا العقد التقني إلى:

  • العقد التجاري الموقّع بين الطرفين بقيمة 40,000 ريال سعودي.
  • وثيقة الخطة التنفيذية المعتمدة (4 مراحل / 4 شهور).
  • وثيقة التحليل التقني الشامل المُسلَّمة للمبرمج.
  • نظام حماية البيانات الشخصية السعودي (PDPL).
  • متطلبات TVTC لاعتماد المعاهد التدريبية.
  • معايير OWASP Top 10 لأمن تطبيقات الويب.
  • معايير WCAG 2.1 AA لإمكانية الوصول.
📌 ملاحظة هامة

في حال تعارض أي بند بين هذا العقد التقني والعقد التجاري، تسود أحكام العقد التجاري في الجوانب المالية والقانونية، ويسود هذا العقد التقني في الجوانب الفنية والتشغيلية.

📦 المرحلة 1: الأساس والاختبارات

  • Workflow Engine (XState)

معايير الجودة

يلتزم الطرف الثاني بمعايير الجودة التالية في كل مرحلة:

🧪 الاختبارات

  • تغطية اختبارات الوحدة (Unit Tests) ≥ 80٪ للمنطق التجاري.
  • اختبارات تكامل (Integration Tests) لكل Server Action رئيسي.
  • اختبارات End-to-End (E2E) لـ 7 سيناريوهات قبول.
  • اختبارات أمنية: npm audit + Snyk قبل كل تسليم.

💎 جودة الكود

  • الالتزام بـ TypeScript strict mode + منع any.
  • تطبيق ESLint + Prettier على كل ملف.
  • اتباع Conventional Commits في رسائل الـGit.
  • Code Reviews قبل الدمج لكل PR.
  • توثيق JSDoc لكل دالة عامة.

🎨 معايير الواجهة

  • الالتزام بـ WCAG 2.1 AA (إمكانية الوصول).
  • دعم RTL عربي أصلي (Logical CSS Properties).
  • تجربة موحّدة على Desktop + Laptop + Tablet.
  • Lighthouse Score ≥ 90 لكل صفحة.

المادة 6: الأمان والامتثال

06
6
الأمان والامتثال التنظيمي

يلتزم الطرف الثاني بمعايير الأمان والامتثال التالية:

🔒 PDPL — نظام حماية البيانات الشخصية

  • تشفير البيانات في النقل TLS 1.3 وفي التخزين AES-256.
  • حق الطالب في الوصول والتصحيح والمحو والتصدير لبياناته.
  • سجل معالجة البيانات (Processing Register) محدّث.
  • إشعار الطرف الأول بأي اختراق محتمل خلال 72 ساعة.
  • الاحتفاظ بالبيانات داخل المملكة (في مرحلة الإنتاج النهائية).

🛡️ الأمان التطبيقي (Application Security)

  • الحماية من ثغرات OWASP Top 10 (SQL Injection، XSS، CSRF، إلخ).
  • Row Level Security (RLS) على كل جدول في قاعدة البيانات.
  • 2FA إلزامي للأدوار: Super Admin، Admin، Finance، Branch Manager.
  • Lockout policy: 5 محاولات فاشلة → قفل 15 دقيقة.
  • Rate Limiting على Endpoints الحساسة (تسجيل دخول، تصدير، بحث).
  • CAPTCHA (Cloudflare Turnstile) عند المحاولات المتكررة.
  • Security Headers كاملة (HSTS، CSP، X-Frame-Options).

📋 Audit Log

  • تسجيل كل العمليات الحساسة (تغيير الحالات، فك الحجب، تعديل الدرجات).
  • السجل غير قابل للحذف لمدة سنتين.
  • واجهة للأدمن للبحث في السجل.

الصيانة والدعم الفني

فترة الضمان والصيانة المجانية 12 شهراً كاملاً تبدأ من تاريخ التسليم النهائي، وتشمل المعايير التالية:

⏱ تصنيف الأعطال ومستويات الاستجابة

الدرجة الوصف زمن الاستجابة زمن الحلّ
🔴 حرجة النظام متوقّف / ميزة رئيسية معطّلة 1 ساعة 4 ساعات
🟠 عالية ميزة ثانوية معطّلة / خطأ يعوّق العمل 4 ساعات 24 ساعة
🟡 متوسطة خطأ لا يعوّق العمل بشكل كبير 24 ساعة 72 ساعة
🟢 منخفضة تحسينات بصرية أو نصية 3 أيام عمل أسبوع

📞 قنوات الدعم الفنّي

  • البريد الإلكتروني: الاستفسارات والطلبات غير العاجلة.
  • WhatsApp Business: العاجل والمستعجل.
  • اجتماع شهري: 30 دقيقة لاستعراض الأداء.
  • تقرير شهري: ملخّص الأداء + الإصلاحات + التحسينات.

✅ ما تشمله الصيانة المجانية

  • إصلاح الأخطاء (Bugs) ومشاكل النظام.
  • تحديثات الأمان للمكتبات والإطار البرمجي.
  • مراقبة النظام والاستجابة الاستباقية للمشاكل.
  • التعديلات البسيطة التي لا تتجاوز ساعة عمل.
  • دعم استفسارات المستخدمين عبر القنوات المعتمدة.
⚠️ ما لا تشمله الصيانة المجانية

تطوير ميزات جديدة، تعديلات كبيرة، تكامل مع أنظمة جديدة، الأخطاء الناتجة عن إساءة الاستخدام أو تعديل الكود من قِبَل طرف آخر. تخضع هذه لإجراءات طلب التغيير (المادة 8).

إدارة التغيير والإصدارات

🌿 استراتيجية Git

  • main — الإصدار الإنتاجي.
  • develop — التطوير الجاري.
  • feature/* — الميزات الجديدة.
  • hotfix/* — الإصلاحات الحرجة.
  • الإصدارات بصيغة Semantic Versioning (مثال: v1.2.3).

📝 طلبات التغيير (Change Requests)

لأي ميزة جديدة أو تعديل خارج نطاق العقد الأصلي:

  • تقديم طلب كتابي يصف التغيير المطلوب.
  • الطرف الثاني يُقدّم تقديراً للجهد والمدّة خلال 5 أيام عمل.
  • الموافقة الكتابية من الطرف الأول قبل البدء.
  • توثيق التغيير في ملحق (Addendum) يُلحق بالعقد.

🚀 النشر (Deployment)

  • نشر تلقائي عبر GitHub Actions.
  • بيئتان: Staging (للاختبار) + Production (للإنتاج).
  • كل دمج لـmain يحتاج موافقة + اجتياز جميع الاختبارات.
  • Rollback تلقائي عند فشل الاختبارات.

معايير القبول الفني

تُعتمد كل مرحلة بعد تجاوزها معايير القبول التالية:

  • جميع الميزات الموضّحة في المخرَجات الفنية مكتملة وتعمل.
  • جميع اختبارات Unit + Integration + E2E تمرّ بنجاح.
  • تغطية الاختبارات ≥ 80٪ للمنطق التجاري.
  • Lighthouse Performance ≥ 90، Accessibility ≥ 95.
  • لا توجد ثغرات أمنية حرجة أو عالية (Snyk + npm audit).
  • RLS Policies مُختبَرة لكل الأدوار.
  • Audit Log يعمل لكل العمليات الحساسة.
  • التوثيق المطلوب مكتمل ومُسلَّم.
  • التدريب مُنفَّذ بنجاح.
  • اجتياز UAT (User Acceptance Testing) من الطرف الأول.
